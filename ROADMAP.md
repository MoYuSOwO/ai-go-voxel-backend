# 体素游戏后端开发路线图 V4.0 - 基于 Zinx 与 Protobuf 的开发流程

## 概述
本路线图基于 **MVP（Minimum Viable Product）** 边界定义，将开发过程拆分为 **4 个工作量均衡的里程碑**。每个里程碑完成后都应进行功能验证，确保核心数据流闭环的完整性。

**本版本核心变更**：
1. **网络层框架化**：使用 Zinx 框架替代原生 TCP 手搓实现
2. **协议标准化**：使用 Protobuf 定义所有通信协议，替代手写二进制解析
3. **开发流程优化**：利用 Zinx 和 Protobuf 的成熟工具链加速开发

**MVP 核心目标**：
```
玩家进游戏 → 走动并同步位置 → 异步加载和卸载视野内区块 → 放置与挖掘方块并同步 → 区块数据异步存盘
```

## 里程碑 1：Zinx 框架集成与 Protobuf 协议定义
**目标**：搭建基于 Zinx 的网络层框架，定义 Protobuf 通信协议，实现 World 容器基础架构。

**工作量估算**：~25% 总工作量

### 核心任务
1. **Zinx 框架集成**
   - 引入 Zinx 依赖（`github.com/aceld/zinx`）
   - 配置 Zinx 全局设置（端口、最大连接数、WorkerPool 等）
   - 实现服务器启动与优雅关闭
   - 设置连接 Hook（连接建立/断开回调）

2. **Protobuf 协议定义**
   - 创建 `.proto` 文件定义所有数据包（Login、Move、ChunkData 等）
   - 配置 Protobuf 编译流程，生成 Go 代码
   - 定义消息 ID 与 Zinx MsgID 的映射关系
   - 编写协议版本兼容性规范

3. **Zinx Router 实现（搬运工模式）**
   - 实现 `LoginRouter`、`MoveRouter`、`BlockRouter` 等基础路由
   - 严格遵循"搬运工"原则：仅解析 Protobuf → 封装事件 → 发送到 World Channel
   - 实现错误处理：Protobuf 解析失败立即断开连接
   - 添加 Channel 发送超时机制，防止 Router 阻塞

4. **World 容器适配**
   - 修改 `PacketEvent` 结构，支持 Protobuf 消息类型
   - 创建 `ConnectionEvent` 通道，处理连接建立/断开
   - 实现 `BroadcastEvent` 结构，支持 Protobuf 消息广播
   - 保持原有 Tick 循环设计，仅修改网络事件处理接口

5. **基础实体架构**
   - 定义 `Entity` 基础结构体（ID、Type、Position）
   - 实现 `EntityManager` 接口框架
   - 创建 `Player` 结构体（嵌入 Entity + Connection 组件）

### 验收标准
- ✅ Zinx 服务器能启动并监听指定端口
- ✅ Protobuf 编译成功，生成正确的 Go 代码
- ✅ 客户端发送 Protobuf 消息后，Zinx Router 能正确解析并转发到 World Channel
- ✅ World 容器能接收并处理 PacketEvent（打印日志即可）
- ✅ 连接建立/断开能正确触发 Hook 并通知 World 容器
- ✅ 通过 `go test -race` 所有单元测试，无数据竞争
- ✅ 严格遵循"Router 零业务逻辑"原则，代码审查通过

### 技术风险与缓解
- **风险**：Zinx 框架与现有架构不兼容
- **缓解**：提前进行技术验证，创建最小化集成示例
- **风险**：Protobuf 序列化性能未知
- **缓解**：进行性能基准测试，对比原有手写二进制解析
- **风险**：Channel 通信导致死锁
- **缓解**：使用带缓冲的通道，明确 Channel 的创建者与关闭责任

## 里程碑 2：KV 数据存储与区块管理（保持原样）
**目标**：集成高性能 KV 数据库，实现区块的异步加载/卸载框架，建立引用计数机制。

**重要说明**：本里程碑基本保持原样，仅需微调以适应 Protobuf 的区块数据格式

**工作量估算**：~25% 总工作量

### 核心任务
1. **KV 数据库集成**
   - 选择并集成存储引擎（BadgerDB 优先，备选 bbolt）
   - 设计数据库 Schema：区块 Key 编码、实体 Key 编码
   - 实现数据库连接池与优雅关闭

2. **区块序列化设计（适配 Protobuf）**
   - 实现 `chunkKey(x, z int32) uint64` 坐标编码
   - 调整区块数据序列化格式，匹配 `proto.ChunkData` 消息结构
   - 集成 zstd 压缩库（`github.com/klauspost/compress/zstd`）
   - 编写序列化/反序列化单元测试，确保与 Protobuf 兼容

3. **异步 ChunkManager 实现**
   - 实现 `ChunkManager` 接口：`LoadAsync`、`UnloadAsync`、`GetChunk`
   - 创建内存 LRU 缓存，限制最大区块数量
   - 实现异步任务队列：加载任务、卸载任务
   - 使用独立 goroutine 处理磁盘 IO

4. **引用计数与状态机**
   - 在 `Chunk` 结构中添加 `ViewerCount int32`（原子操作）
   - 定义区块状态枚举：`Loading`、`Active`、`Unloading`
   - 实现视野追踪基础框架（玩家→区块映射）

5. **简单地形生成**
   - 实现平面地图生成器（所有区块 Y=0 层为草方块）
   - 集成到 ChunkManager 的加载流程中

### 验收标准
- ✅ 数据库集成成功，服务器启动时初始化，关闭时优雅清理
- ✅ 区块能正确序列化为 Protobuf 兼容格式并保存到数据库
- ✅ 从数据库加载区块数据能正确反序列化为 `proto.ChunkData`
- ✅ 异步加载：请求新区块时立即返回，数据就绪后通过回调通知
- ✅ 引用计数：玩家进入/离开视野时，ViewerCount 正确增减
- ✅ ViewerCount=0 的区块能自动标记为待卸载
- ✅ 内存缓存限制有效，超过上限时自动淘汰最久未使用区块
- ✅ 通过 `-race` 测试，验证并发安全性

### 技术风险与缓解
- **风险**：异步回调导致数据竞争
- **缓解**：所有回调都在 World 的主 goroutine 中执行，通过通道传递
- **风险**：zstd 压缩/解压性能瓶颈
- **缓解**：使用低压缩级别，测试百区块同时加载的压力表现
- **风险**：数据库损坏导致数据丢失
- **缓解**：实现定期备份机制，添加数据完整性校验

## 里程碑 3：玩家移动与区块同步（适配 Protobuf）
**目标**：实现玩家移动处理、视野管理、区块异步加载/卸载，完成玩家位置同步。

**工作量估算**：~25% 总工作量

### 核心任务
1. **玩家移动协议处理（Protobuf 版本）**
   - 实现 `PlayerMove` 消息的完整处理流程
   - 在 World 的 tick 循环中处理移动事件
   - 更新玩家位置，检查边界（世界边界、Y轴限制）
   - 验证消息合法性：玩家是否存在、坐标是否有效

2. **视野管理与区块加载**
   - 计算玩家视野范围（以区块为单位，例如半径 5 个区块）
   - 实现玩家→区块的映射关系，管理 ViewerCount
   - 当玩家进入新区块视野时，触发异步加载
   - 当玩家离开区块视野时，减少引用计数

3. **区块数据同步（Protobuf 版本）**
   - 实现 `ChunkData` 消息的构造与发送
   - 区块加载完成后，自动发送给对应玩家
   - 使用 Zinx 的 `SendPacket` 接口发送 Protobuf 消息
   - 支持批量发送，避免单个区块一个包

4. **玩家位置同步（Protobuf 版本）**
   - 实现 `PlayerListUpdate` 消息的构造
   - 在 tick 循环中定期收集玩家位置变化
   - 根据视野范围筛选需要同步的玩家
   - 优化广播范围，只发送给相关玩家
   - 使用 Zinx 的 `BroadcastToPlayers` 接口进行高效广播

5. **World 驱动逻辑完善**
   - 完善 tick 循环的顺序：移动→视野更新→区块加载→同步
   - 实现待处理操作队列（移动、方块操作）
   - 添加 World 状态监控（在线玩家数、活跃区块数）

### 验收标准
- ✅ 客户端发送 `PlayerMove` 消息后，服务器能更新玩家位置
- ✅ 玩家移动时，视野内的新区块能自动异步加载并发送 `ChunkData`
- ✅ 玩家离开区块视野后，ViewerCount 减少，长时间无人观看的区块被标记卸载
- ✅ 多个玩家在附近时，能互相看到位置更新（`PlayerListUpdate`）
- ✅ 视野管理正确，不会加载视野外的区块
- ✅ 网络带宽优化：同一区块只向同一玩家发送一次
- ✅ 性能要求：100 玩家同时移动，服务器 CPU 使用率 < 70%
- ✅ 内存管理：活跃区块数稳定，无持续增长

### 技术风险与缓解
- **风险**：玩家快速移动导致区块加载风暴
- **缓解**：实现加载队列优先级，限制每 tick 最大加载数量
- **风险**：位置同步频率过高导致网络拥堵
- **缓解**：降低同步频率（每 3 tick 同步一次），使用差值压缩
- **风险**：视野计算成为性能瓶颈
- **缓解**：使用空间索引（网格分区），优化玩家查找算法

## 里程碑 4：方块操作与数据闭环（适配 Protobuf）
**目标**：实现方块放置/破坏操作，完成脏区块异步存盘，验证 MVP 完整数据流闭环。

**工作量估算**：~25% 总工作量

### 核心任务
1. **方块操作协议处理（Protobuf 版本）**
   - 实现 `PlaceBlock` 和 `BreakBlock` 消息的解析与处理
   - 验证操作合法性：坐标是否在已加载区块内
   - 在 World 中处理方块操作，更新区块数据
   - 立即同步方块变化给视野内的其他玩家

2. **区块数据修改与同步**
   - 实现区块内方块数据的线程安全修改
   - 标记修改后的区块为"脏"（需要存盘）
   - 使用 Protobuf 消息格式同步方块变化
   - 支持增量更新（可选优化）

3. **脏区块异步存盘**
   - 实现脏区块检测机制（每 tick 或定时）
   - 将脏区块加入异步存盘队列
   - 在独立 goroutine 中执行压缩、序列化、存盘
   - 存盘完成后清除脏标记

4. **数据一致性验证**
   - 实现服务器重启后数据恢复测试
   - 验证区块修改后存盘，重启后数据不丢失
   - 测试多玩家同时修改同一区块的并发安全性

5. **MVP 端到端测试（Zinx + Protobuf 版本）**
   - 创建完整测试场景：登录→移动→加载区块→放置方块→退出→重启→验证
   - 模拟多玩家并发操作（10+ 玩家同时在线）
   - 性能压测：验证在负载下数据流闭环稳定性

6. **监控与调试工具**
   - 添加 metrics 导出：TPS、在线玩家、区块加载数、存盘队列长度
   - 监控 Zinx 连接数、WorkerPool 状态
   - 实现基础管理命令（通过 TCP 或控制台）
   - 添加详细日志，便于问题排查

### 验收标准
- ✅ 客户端能放置和破坏方块，操作立即生效并通过 Protobuf 消息同步给其他玩家
- ✅ 方块修改后，对应区块被标记为脏，并在后台异步存盘
- ✅ 服务器重启后，之前修改的方块数据仍然存在
- ✅ 多玩家同时修改相邻方块，无数据竞争或丢失
- ✅ 脏区块存盘不影响主 tick 性能（存盘延迟 < 100ms）
- ✅ 完整 MVP 流程测试通过：从登录到方块操作到存盘的全链路
- ✅ 性能要求：50 玩家同时在线并频繁操作，服务器稳定运行 1 小时
- ✅ Zinx 框架监控指标可用，能实时查看连接状态和消息处理效率

### 技术风险与缓解
- **风险**：脏区块存盘延迟导致数据丢失（服务器崩溃）
- **缓解**：实现定期检查点（checkpoint），减少数据丢失窗口
- **风险**：多玩家同时修改同一方块导致逻辑冲突
- **缓解**：使用乐观锁或最后写入获胜策略，记录操作时序
- **风险**：存盘队列堆积导致内存增长
- **缓解**：实现背压机制，当队列过长时暂停新操作

## 里程碑完成后的 MVP 状态
完成所有 4 个里程碑后，我们将拥有一个 **完整的体素游戏后端 MVP**，具备以下能力：

### 核心功能
1. **玩家管理**：基于 Zinx 的连接管理，Protobuf 登录协议
2. **位置同步**：玩家移动实时同步给视野内其他玩家，使用 Protobuf 消息格式
3. **区块管理**：基于视野的异步加载/卸载，引用计数自动清理
4. **方块操作**：放置/破坏方块，通过 Protobuf 实时同步，数据持久化
5. **数据持久化**：所有区块修改异步存盘到 KV 数据库，支持服务器重启恢复

### 性能指标（目标）
- 支持 **100+ 玩家** 同时在线（Zinx 框架优化）
- 游戏刻间隔 **50ms**（20 TPS）
- 区块加载延迟 **< 200ms**（从请求到收到数据）
- 内存占用 **< 2GB**（100 玩家，2000 活跃区块）
- 网络带宽 **< 10 Mbps**（100 玩家平均移动）

### 技术栈优势
- **网络层**：Zinx 提供稳定、高性能的 TCP 通信基础
- **协议层**：Protobuf 提供清晰、版本兼容的消息定义
- **存储层**：BadgerDB 提供高性能 KV 存储，zstd 提供高效压缩
- **逻辑层**：保持原有的异步并发设计，经过验证的稳定性

## 后续规划（MVP 之后）
完成 MVP 后，可根据需求优先级开发以下功能：

### 高优先级
1. **物理系统**：重力、碰撞检测、跳跃
2. **物品系统**：基础背包、物品拾取、手持物品
3. **生物系统**：简单怪物 AI、生成与清理

### 中优先级
1. **世界生成**：噪声地形、洞穴、矿物分布
2. **游戏模式**：创造/生存模式切换
3. **聊天系统**：玩家间文字通信（可通过 Protobuf 扩展消息类型）

### 低优先级
1. **红石系统**：基础电路逻辑
2. **天气系统**：雨雪、昼夜循环
3. **成就系统**：数据统计与成就解锁
